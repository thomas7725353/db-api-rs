# React QueryBuilder Frontend Design

## Goal

Replace the bundled DBAPI 3.3.0 Vue static UI with a maintainable React frontend and make QueryBuilder the primary API creation path. The new UI targets the Rust standalone runtime only.

## Scope

In scope:

- Add a new React/Vite/TypeScript frontend under `runtime-rust/frontend`.
- Use `react-querybuilder` and `@react-querybuilder/antd` for visual query rule editing.
- Use Ant Design for application UI components.
- Use Tailwind CSS only for layout and utility styling where it does not fight Ant Design.
- Build the frontend into `runtime-rust/static` so Axum keeps serving a single app on port `8520`.
- Support the Rust standalone core flows:
  - datasource list/create/edit/delete/connect
  - API list/search/tree/create/edit/delete/online/offline
  - SQL API mode for compatibility
  - QueryBuilder API mode for ordinary users
  - request testing with Authorization token
  - app/token creation and group authorization
  - access log monitor table and basic summary
- Store QueryBuilder API definitions in existing metadata:
  - `api_sql.sql_text` contains DSL JSON
  - `api_sql.transform_plugin = "queryBuilder"`

Out of scope for the first frontend rewrite:

- Nacos or cluster mode.
- Firewall settings.
- Import/export/download docs endpoints that are not implemented in the Rust runtime.
- Full visual chart parity with old ECharts monitor.
- A schema migration for a new `api_config.engine` column.

## Frontend Architecture

The React app lives in `runtime-rust/frontend` and is built by Vite. It uses React Router for routes, Ant Design for widgets, and a small API client wrapper around `fetch`. The build output is copied to `runtime-rust/static`.

Route groups:

- `/datasources`: datasource list and editor modal.
- `/apis`: API list, group filter, online/offline actions.
- `/apis/new` and `/apis/:id/edit`: API editor with SQL or QueryBuilder mode.
- `/apis/:id/request`: request tester.
- `/tokens`: app/token management and group authorization.
- `/monitor`: access log and basic metrics.

The UI will not try to clone the old Vue app visually. It should preserve behavior, endpoint compatibility, and route fallbacks while using a cleaner single-page React layout.

## QueryBuilder API Model

The API editor supports `engine = "sql" | "queryBuilder"`.

SQL mode stores normal SQL in `api_sql.sql_text` and leaves `transform_plugin` empty or `"sql"`.

QueryBuilder mode stores JSON in `api_sql.sql_text` and sets `transform_plugin = "queryBuilder"`.

Stored JSON:

```json
{
  "type": "queryBuilder",
  "table": "demo_item",
  "select": ["id", "name", "status", "price", "note"],
  "rules": {
    "combinator": "and",
    "rules": [
      {
        "field": "status",
        "operator": "in",
        "valueSource": "param",
        "value": "statusList",
        "defaultValue": ["active", "pending"],
        "skipWhen": ["missing", "empty_array"]
      },
      {
        "field": "price",
        "operator": ">",
        "valueSource": "param",
        "value": "minPrice",
        "defaultValue": 100,
        "skipWhen": ["missing"]
      }
    ]
  },
  "orderBy": [{ "field": "id", "direction": "desc" }],
  "limit": { "param": "limit", "default": 20, "max": 100 },
  "offset": { "param": "offset", "default": 0 },
  "count": true
}
```

`valueSource = "param"` means runtime requests provide the value. `valueSource = "value"` means the rule contains a fixed constant.

Missing, `null`, empty string, empty array, `0`, and `false` are distinct. A condition is skipped only when its `skipWhen` list explicitly matches the runtime input state.

## Backend Contracts Needed

The frontend depends on existing endpoints first:

- `/datasource/getAll`, `/datasource/add`, `/datasource/update`, `/datasource/delete/{id}`, `/datasource/connect`
- `/apiConfig/getAll`, `/apiConfig/search`, `/apiConfig/detail/{id}`, `/apiConfig/add`, `/apiConfig/update`, `/apiConfig/delete/{id}`, `/apiConfig/online/{id}`, `/apiConfig/offline/{id}`
- `/group/getAll`, `/group/create`, `/group/delete/{id}`
- `/app/create`, `/app/getAll`, `/app/delete/{id}`, `/app/auth`, `/app/getAuthGroups/{id}`
- `/token/generate`
- `/access/search`, `/access/countByDay`, `/access/successRatio`
- `/system/version`, `/system/mode`, `/system/getIPPort`, `/system/getIP`

QueryBuilder backend endpoints are planned after the UI skeleton:

- `/queryBuilder/parse`: validate DSL, return inferred runtime params, and optionally preview SQL.
- `/queryBuilder/execute`: execute a DSL against a datasource for editor testing.
- `/api/{path}`: if `transform_plugin = "queryBuilder"`, compile the stored DSL with request params and execute via SeaQuery/SeaORM.

## Safety Rules

- The browser never sends SQL generated by QueryBuilder to be trusted directly.
- The backend validates table names, selected columns, condition fields, and order fields.
- Operators are allowlisted.
- User request values are always bind parameters.
- Request parameters cannot change table name, field name, selected columns, or operator.
- `limit.max` caps runtime limit.

## Build And Deployment

Local development:

- `cd runtime-rust/frontend`
- `npm install`
- `npm run dev`

Production build:

- `npm run build`
- output: `runtime-rust/static`

Docker build gains a Node build stage before the Rust runtime image copies `runtime-rust/static`.

## Testing

Frontend:

- TypeScript build passes.
- Vite production build writes `runtime-rust/static/index.html`.
- API list loads from the Rust runtime.
- API editor can save SQL mode and QueryBuilder mode.
- Request tester can call a token-protected API.

Backend integration after QueryBuilder compiler:

- QueryBuilder DSL validates.
- QueryBuilder API returns rows.
- Token auth and access logging still work.
