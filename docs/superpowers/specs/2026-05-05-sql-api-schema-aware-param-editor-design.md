# SQL API Schema-Aware Param Editor Design

## Context

The current React API editor supports two engines:

- QueryBuilder mode already loads real datasource tables and columns through `/table/getAllTables` and `/table/getAllColumns`.
- SQL mode still uses a free SQL textarea plus a free-form request parameter table, so users can mistype field names or choose wrong parameter types.

The immediate target is the four seeded Demo CRUD SQL APIs:

- `demo/items/get`
- `demo/items/create`
- `demo/items/update`
- `demo/items/delete`

The implementation should be generic for SQL APIs, but the seed data and default editing behavior must make those four APIs hard to misconfigure.

## Approaches Considered

### Recommended: Schema-Aware SQL Editor With CRUD Guardrails

Add table selection and schema-derived parameter editing to SQL mode. Users pick a table after selecting a datasource. Parameter names are selected from real table columns. Parameter types are derived from backend column metadata and mapped to DBAPI parameter types by the frontend. Demo CRUD seeds are tightened so create/update/get/delete use the expected parameters.

This reuses the QueryBuilder metadata endpoints and keeps the SQL editor compatible with existing free SQL text.

### Alternative: Convert Demo CRUD To QueryBuilder

Move all Demo CRUD APIs to QueryBuilder-style structured definitions. This would give stronger identifier safety, but QueryBuilder currently focuses on select/list APIs and does not model insert/update/delete well enough.

### Alternative: Backend-Only SQL Parser Inference

Try to parse SQL and infer parameters, table names, and column types server-side. This is more robust long-term, but it is too broad for this step and would need dialect-aware parsing for SQLite/MySQL/Postgres.

## Selected Design

Use the recommended approach.

SQL mode keeps the raw SQL textarea. Above or beside it, add a table selector that loads from the selected datasource. Once a table is selected, load columns from `/table/getAllColumns`. The request parameter editor becomes schema-aware:

- Parameter name is a dropdown of table columns.
- Parameter type is read-only and auto-filled from the selected column type.
- Description remains editable.
- The add button is hidden when an API has a fixed parameter contract.

Existing APIs without a selected table should still be editable, but they get a warning and fall back to the current manual parameter behavior only if schema metadata is unavailable.

## CRUD Rules

The Demo CRUD API seeds should be updated as follows:

- `get`: SQL is `SELECT ... FROM demo_items WHERE id = $id`; params are fixed to `id`; no add/delete controls.
- `delete`: SQL is `DELETE FROM demo_items WHERE id = $id`; params are fixed to `id`; no add/delete controls.
- `create`: SQL inserts only business fields. It must not include `id`, `created_at`, or `updated_at` as request parameters. `created_at` and `updated_at` are generated by database default or SQL expression.
- `update`: SQL updates business fields by primary key. Params include `id`, `name`, `status`, and `note`; `created_at` and `updated_at` are not user parameters. `updated_at` is generated by database default/SQL expression.

For this product assumption, every table has `id` as the primary key. The frontend can therefore treat `id` as the locked primary-key field for single-row get/update/delete templates.

## Type Mapping

Backend column types are database-specific strings. The frontend maps them into DBAPI parameter types:

- integer-like: `bigint`
- float/decimal/real/numeric-like: `double`
- date/time-like: `date`
- everything else: `string`

Array parameter types are not inferred from table columns in SQL mode. They remain a separate manual feature for future list filters.

## Data Flow

1. User selects datasource.
2. SQL editor loads tables with `tableService.tables(datasourceId)`.
3. User selects table.
4. SQL editor loads columns with `tableService.columns(datasourceId, table)`.
5. Param editor uses the loaded columns as dropdown options.
6. Selecting a column updates the parameter name and read-only mapped type.
7. Saving persists `api_config.params` as JSON exactly as today.

No backend schema changes are needed.

## Error Handling

- If datasource is missing, table and field dropdowns are disabled.
- If table loading fails, show a warning and keep the SQL textarea usable.
- If column loading fails, show a warning and keep existing params visible.
- If a saved parameter references a column no longer present, keep the row visible with a warning-style label instead of silently deleting it.

## Testing

Frontend checks:

- `npm run build` from `runtime-rust/frontend`.
- Verify SQL editor compiles with new props and type mapping helpers.

Runtime checks:

- `cargo test` from `runtime-rust` if backend seed or behavior changes.
- Re-seed Demo CRUD if needed, then verify `demo_item_create/get/update/delete` rows in `data.db`.
- Browser smoke test: edit each of the four Demo CRUD APIs and confirm:
  - get/delete show fixed `id` with type `bigint` and no add button.
  - create excludes `id`, `created_at`, and `updated_at`.
  - update includes `id` and business fields only.
  - field dropdown options come from `demo_items` schema.

## Omission Checklist

- Do not make parameter type editable when a field is selected.
- Do not let get/delete add extra parameters.
- Do not put `created_at` into create request params.
- Do not put `created_at` or `updated_at` into update request params.
- Do not remove the raw SQL textarea.
- Do not require QueryBuilder for insert/update/delete.
- Do not depend on static `demo_items` field names outside the seed-specific defaults.
